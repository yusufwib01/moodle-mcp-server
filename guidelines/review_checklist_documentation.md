# Review Checklist — Documentation (Moodle)

Reference: Peer review category 13.

## PHPDoc

- Every public function/method has a PHPDoc block.
- Description explains *what* and *why*, not the obvious "what".
- `@param` for every argument with the actual type.
- `@return` present whenever return is non-void.
- `@since` added when introducing a new function in a specific release.
- `@deprecated since X.Y` accompanies any deprecation (also call `debugging()`).

## upgrade.txt

- Component-level `upgrade.txt` updated for behaviour changes, schema changes, removed/renamed APIs, new hooks.
- Entries under the bumped version header in the same patch.
- Wording: "Function X has been deprecated. Use Y instead." (action-oriented).

## Tracker labels

- `docs_required` — applied if user-facing or developer-facing docs need updating outside the repo.
- `ui_change` — applied if any user-visible UI changed.
- `affects_mobileapp` — applied if mobile app behaviour or web services change.

## Component library / dev docs

- New UI components documented in the component library (since 4.0).
- New public APIs have moodledev.io page references where relevant.

## Common review findings

- PHPDoc autogenerated by IDE with empty descriptions.
- `upgrade.txt` skipped because "it's just a fix" — even small public-API tweaks belong here.
- Missing `ui_change` on a patch that adds a new dialog.
